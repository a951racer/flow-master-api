# Requirements Document

## Introduction

This feature defines a RESTful API built with Node.js, Express, and MongoDB. The API manages six primary resources: **Users** (for authentication), **Expenses** (for tracking financial obligations), **ExpenseCategories** (for classifying expenses), **PaymentSources** (for identifying payment methods), **Incomes** (for tracking recurring income entries), and **Periods** (for representing pay periods). Periods embed their associated expense and income entries as subdocuments. It provides standard CRUD operations over HTTP, with structured request/response handling, input validation via Zod, JWT-based authentication, error management, and database connectivity via MongoDB. The system is designed to serve as a backend service for client applications.

## Glossary

- **API**: The RESTful HTTP service built with Node.js and Express that handles client requests and returns JSON responses.
- **Client**: Any application or user agent that sends HTTP requests to the API.
- **User**: A domain entity representing a registered account, identified by email address, with a bcrypt-hashed password stored in MongoDB.
- **Expense**: A domain entity representing a financial obligation (expense, debt, or bill), persisted in MongoDB. Expenses are global and accessible to all authenticated users; they are not scoped to any individual User.
- **Collection**: A MongoDB collection that stores documents for a given resource type.
- **Document**: A single MongoDB record representing one instance of a User or Expense.
- **Router**: The Express component responsible for mapping HTTP routes to handler functions.
- **Controller**: The module that processes a request, invokes the appropriate service logic, and returns a response.
- **Service**: The module that contains business logic and interacts with the Repository.
- **Repository**: The module responsible for all MongoDB read and write operations.
- **Validator**: The module that uses Zod schemas to check incoming request payloads and route parameters.
- **Zod**: The TypeScript-first schema declaration and validation library used to define and enforce all request schemas.
- **JWT**: JSON Web Token — a signed, self-contained token used to authenticate and authorize API requests.
- **JWT_Authenticator**: The Express middleware that validates the JWT on protected routes and attaches the decoded payload to the request context.
- **Token_Service**: The module responsible for signing and verifying JWTs.
- **Error_Handler**: The Express middleware that catches errors and returns structured error responses.
- **Request**: An incoming HTTP message received by the API.
- **Response**: An outgoing HTTP message sent by the API to the Client.
- **ExpenseType**: An enumeration of expense classifications: `"expense"`, `"debt"`, or `"bill"`.
- **ExpenseCategory**: A domain entity representing a named expense classification, persisted as a Document in its own MongoDB collection. Each ExpenseCategory has a single `category` string field (max 100 characters). Expenses reference an ExpenseCategory by its MongoDB ObjectId.
- **PaymentSource**: A domain entity representing a named payment method, persisted as a Document in its own MongoDB collection. Each PaymentSource has a single `source` string field (max 100 characters). Expenses reference a PaymentSource by its MongoDB ObjectId.
- **ObjectId**: MongoDB's native 24-character hexadecimal unique identifier, auto-generated by Mongoose on Document creation. Exposed as `_id` in all API responses.
- **BaseDocument**: The set of fields automatically managed by Mongoose that are present on every Document: `_id`, `createdAt`, and `updatedAt`. These fields are never settable by the Client.
- **Income**: A domain entity representing a recurring income entry, persisted as a Document in its own MongoDB `incomes` Collection. Income is global and accessible to all authenticated users; it is not scoped to any individual User.
- **Period**: A domain entity representing a pay period, persisted as a Document in its own MongoDB `periods` Collection. Period is global and accessible to all authenticated users; it is not scoped to any individual User. Each Period embeds an array of **PeriodExpenseEntry** subdocuments and an array of **PeriodIncomeEntry** subdocuments.
- **PeriodExpenseEntry**: An embedded subdocument within a Period that records the payment status and optional override amount for one Expense within that Period. Not stored in a separate collection.
- **PeriodIncomeEntry**: An embedded subdocument within a Period that records the receipt status and optional override amount for one Income within that Period. Not stored in a separate collection.
- **PeriodExpenseStatus**: An enumeration of payment statuses for a PeriodExpenseEntry: `"Unpaid"`, `"Paid"`, or `"Deferred"`.
- **PeriodIncomeStatus**: An enumeration of receipt statuses for a PeriodIncomeEntry: `"Pending"` or `"Received"`.
- **helmet**: An Express middleware package that sets secure HTTP response headers (e.g. `X-Content-Type-Options`, `X-Frame-Options`, `Strict-Transport-Security`, `X-XSS-Protection`) to protect against common web vulnerabilities.
- **morgan**: An HTTP request logger middleware for Node.js/Express that logs method, URL, status code, and response time for every incoming request.
- **cors**: An Express middleware package that enables Cross-Origin Resource Sharing (CORS), allowing the API to accept requests from web applications hosted on different domains.
- **Procfile**: A plain-text file at the project root that declares the command Heroku uses to start the web process. For this API the content is `web: node dist/server.js`.
- **Heroku Config Vars**: Environment variables configured in the Heroku dashboard (or CLI) that are injected into the dyno at runtime. They replace the `.env` file in production and are the required mechanism for supplying `MONGODB_URI`, `JWT_SECRET`, `JWT_EXPIRES_IN`, and `NODE_ENV` to the deployed application. `PORT` is set automatically by Heroku and must not be overridden.

## Requirements

### Requirement 0: Base Document Fields

**User Story:** As a developer, I want all MongoDB Documents to include standard metadata fields, so that every record has a consistent, auto-managed identity and audit trail.

#### Acceptance Criteria

1. THE API SHALL configure all Mongoose schemas with `{ timestamps: true }` so that `createdAt` and `updatedAt` are automatically populated by Mongoose on every Document.
2. THE API SHALL rely on MongoDB to auto-generate the `_id` field (ObjectId) for every Document on creation; THE API SHALL NOT manually assign `_id` values.
3. THE Validator SHALL strip or reject `_id`, `createdAt`, and `updatedAt` fields from all request bodies on `POST` and `PUT` endpoints so that Clients cannot set or override these fields.
4. THE API SHALL include `_id`, `createdAt`, and `updatedAt` in all Document responses returned to the Client.

### Requirement 1: Project Structure and Server Initialization

**User Story:** As a developer, I want a well-structured Express application, so that the codebase is maintainable and easy to extend.

#### Acceptance Criteria

1. THE API SHALL expose an HTTP server on a configurable port defined via an environment variable.
2. WHEN the server starts successfully, THE API SHALL log the port number it is listening on.
3. THE API SHALL read `PORT` exclusively from the environment variable provided at runtime; THE API SHALL NOT hardcode a fallback port value in production — when deployed on Heroku, `PORT` is assigned dynamically by the platform and must be used as-is.
4. THE API SHALL load environment variables from a `.env` file at startup (for local development only).
5. THE API SHALL parse incoming JSON request bodies automatically for all routes.
6. THE API SHALL configure CORS (Cross-Origin Resource Sharing) middleware to allow requests from the frontend application.
7. THE API SHALL read the allowed frontend origin from an environment variable named `FRONTEND_URL`; if not set, THE API SHALL default to allowing all origins (`*`) for development purposes.
8. THE API SHALL enable credentials support in CORS configuration to allow cookies and authorization headers to be sent cross-origin.

### Requirement 24: Heroku Deployment

**User Story:** As a developer, I want the API to be deployable to Heroku, so that it can be hosted and accessed as a live service.

#### Acceptance Criteria

1. THE API SHALL include a `Procfile` at the project root with the content `web: node dist/server.js` to define the Heroku web process.
2. THE API SHALL define a `build` script in `package.json` that compiles TypeScript to the `dist/` directory (e.g. `"build": "tsc"`).
3. THE API SHALL define a `start` script in `package.json` that runs the compiled output: `"start": "node dist/server.js"`.
4. THE API SHALL NOT commit the `.env` file to version control; all environment variables (`MONGODB_URI`, `PORT`, `JWT_SECRET`, `JWT_EXPIRES_IN`, `NODE_ENV`) SHALL be configured as Heroku Config Vars in the Heroku dashboard or CLI.
5. THE API SHALL read `PORT` exclusively from `process.env.PORT`; Heroku assigns this value dynamically and the application must bind to it without any hardcoded fallback.
6. THE API SHALL NOT override `NODE_ENV` — Heroku automatically sets `NODE_ENV=production` and the application should respect that value.

### Requirement 3: User Resource Schema

**User Story:** As a developer, I want a well-defined User schema, so that user data is stored consistently and securely in MongoDB.

#### Acceptance Criteria

1. THE API SHALL define a User Document with the following fields:
   - `_id` — auto-generated MongoDB ObjectId, managed exclusively by MongoDB
   - `createdAt` — timestamp auto-populated by Mongoose on Document creation
   - `updatedAt` — timestamp auto-populated by Mongoose on every Document update
   - `firstName` — required string
   - `lastName` — required string
   - `email` — required string, used as the unique login identifier; THE API SHALL enforce uniqueness on the `email` field at the database level
   - `password` — required string, stored exclusively as a bcrypt hash
2. THE Validator SHALL define a Zod schema for User creation requiring `firstName`, `lastName`, `email`, and `password`; the schema SHALL strip or reject `_id`, `createdAt`, and `updatedAt` if present in the request body.
3. THE Validator SHALL validate `email` as a properly formatted email address on all User create and login requests.
4. THE Validator SHALL validate `password` as a string of at least 8 characters on User creation requests.
5. THE API SHALL NOT store or return plaintext passwords in any response.

### Requirement 4: Expense Resource Schema

**User Story:** As a developer, I want a well-defined Expense schema, so that expense data is stored consistently and can be queried reliably.

#### Acceptance Criteria

1. THE API SHALL define an Expense Document with the following fields:
   - `_id` — auto-generated MongoDB ObjectId, managed exclusively by MongoDB
   - `createdAt` — timestamp auto-populated by Mongoose on Document creation
   - `updatedAt` — timestamp auto-populated by Mongoose on every Document update
   - `dayOfMonth` — required number representing the day of the month the expense is due (1–31)
   - `amount` — required number representing the monetary value of the expense
   - `type` — required ExpenseType enum value (`"expense"`, `"debt"`, or `"bill"`)
   - `payee` — required string identifying the recipient of the payment
   - `payeeUrl` — optional string containing a URL associated with the payee
   - `required` — required boolean indicating whether the expense is mandatory
   - `category` — required MongoDB ObjectId referencing an ExpenseCategory Document in the expense-categories Collection
   - `paymentSource` — required MongoDB ObjectId referencing a PaymentSource Document in the payment-sources Collection
   - `inactive` — required boolean, defaults to `false`, indicating whether the Expense record has been deactivated
   - `inactiveDate` — optional ISO 8601 date string (`YYYY-MM-DD`), set only when `inactive` is `true`; represents the date the Expense record was marked inactive
2. THE Validator SHALL define a Zod schema for Expense creation and update that enforces all field types and constraints listed above, validating `category` and `paymentSource` as non-empty strings in valid MongoDB ObjectId format (24-character hexadecimal strings), `inactive` as a boolean, and `inactiveDate` as an optional valid ISO 8601 date string (`YYYY-MM-DD`) that SHALL only be present when `inactive` is `true`; the schema SHALL strip or reject `_id`, `createdAt`, and `updatedAt` if present in the request body.
3. THE Validator SHALL validate `dayOfMonth` as an integer between 1 and 31 inclusive.
4. THE Validator SHALL validate `amount` as a positive number.
5. THE Validator SHALL validate `payeeUrl`, when provided, as a properly formatted URL string.
6. THE Validator SHALL use Zod's strict mode to reject Expense request bodies containing fields not defined in the Expense schema.

### Requirement 11: JWT Authentication

**User Story:** As a developer, I want API routes to be protected with JWT authentication, so that only authorized clients can access or modify resources.

#### Acceptance Criteria

1. THE API SHALL expose a `POST /api/auth/register` endpoint that accepts `firstName`, `lastName`, `email`, and `password`, creates a new User Document with the password stored as a bcrypt hash, and returns a signed JWT with HTTP status `201`.
2. THE API SHALL expose a `POST /api/auth/login` endpoint that accepts `email` and `password`, verifies the credentials against the stored User Document using the `email` field as the login identifier, and returns a signed JWT with HTTP status `200`.
3. WHEN a login request is received with credentials that do not match any User Document, THE Token_Service SHALL return HTTP status `401` with a descriptive error message.
4. THE Token_Service SHALL sign JWTs using a secret key read exclusively from an environment variable named `JWT_SECRET`.
5. THE Token_Service SHALL set a configurable expiry on all issued JWTs, read from an environment variable named `JWT_EXPIRES_IN`, defaulting to `1h` if the variable is not set.
6. THE JWT_Authenticator SHALL extract the JWT from the `Authorization` header using the `Bearer` scheme on all protected routes.
7. WHEN a protected route is accessed with a valid, non-expired JWT, THE JWT_Authenticator SHALL decode the token payload and attach it to the request context before passing control to the next middleware.
8. IF a protected route is accessed without an `Authorization` header, THEN THE JWT_Authenticator SHALL return HTTP status `401` with a descriptive error message.
9. IF a protected route is accessed with a JWT that is invalid or expired, THEN THE JWT_Authenticator SHALL return HTTP status `401` with a descriptive error message.
10. THE Validator SHALL validate the register request body against a Zod schema requiring `firstName` (non-empty string), `lastName` (non-empty string), `email` (valid email format), and `password` (string of at least 8 characters).
11. THE Validator SHALL validate the login request body against a Zod schema requiring `email` (valid email format) and `password` (non-empty string).
12. THE API SHALL store user passwords as bcrypt hashes and SHALL NOT store plaintext passwords in the database.

### Requirement 30: Cascade Expense and Income Activation to Active Periods

**User Story:** As a developer, I want creating an active expense or income, or reactivating one, to automatically add it to all relevant active Periods, so that period views stay current without manual intervention.

#### Acceptance Criteria

1. WHEN an Expense Document is created with `inactive` not set to `true`, THE API SHALL add a PeriodExpenseEntry subdocument (with `status: "Unpaid"`) to every active Period whose date range contains the expense's `dayOfMonth`.
2. WHEN an Expense Document is updated and `inactive` changes from `true` to `false`, THE API SHALL add a PeriodExpenseEntry subdocument (with `status: "Unpaid"`) to every active Period whose date range contains the expense's `dayOfMonth`.
3. WHEN an Income Document is created with `inactive` not set to `true`, THE API SHALL add a PeriodIncomeEntry subdocument (with `status: "Pending"`) to every active Period whose date range contains the income's `dayOfMonth`.
4. WHEN an Income Document is updated and `inactive` changes from `true` to `false`, THE API SHALL add a PeriodIncomeEntry subdocument (with `status: "Pending"`) to every active Period whose date range contains the income's `dayOfMonth`.
5. AN active Period is defined as a Period whose `endDate` is greater than or equal to today's date (UTC).
6. A `dayOfMonth` value is considered to fall within a period if, in any calendar month that overlaps the period, the clamped day (min of `dayOfMonth` and last day of that month) falls on or between `startDate` and `endDate`.
7. IF the expense or income is already present in a period's subdocument array, THE API SHALL NOT add a duplicate entry.
8. Periods whose `endDate` is before today SHALL NOT be modified.

4. Periods whose `endDate` is before today SHALL NOT be modified by the cascade.
5. IF the income was already `inactive: true` before the update, no cascade SHALL occur.

### Requirement 26: Expense Audit Trail

6. THE audit trail SHALL be stored in a dedicated `expense-audits` MongoDB collection and SHALL NOT modify the Expense Document itself.
7. THE `changedAt` field SHALL record the exact date and time of the operation as a UTC timestamp.
8. FOR ObjectId reference fields (`category`, `paymentSource`), THE audit trail SHALL record their string representation for comparison and storage.

### Requirement 17: Period Resource Schema

**User Story:** As a developer, I want a well-defined Period schema, so that pay period data is stored consistently and can be queried reliably.

#### Acceptance Criteria

1. THE API SHALL define a Period Document with the following fields:
   - `_id` — auto-generated MongoDB ObjectId, managed exclusively by MongoDB
   - `createdAt` — timestamp auto-populated by Mongoose on Document creation
   - `updatedAt` — timestamp auto-populated by Mongoose on every Document update
   - `startDate` — required ISO 8601 date string (`YYYY-MM-DD`) representing the first day of the pay period
   - `endDate` — required ISO 8601 date string (`YYYY-MM-DD`) representing the last day of the pay period; SHALL be strictly after `startDate`
   - `expenses` — array of PeriodExpenseEntry subdocuments (default empty), each containing:
     - `expense` — required MongoDB ObjectId referencing an Expense Document
     - `status` — required PeriodExpenseStatus enum value (`"Unpaid"`, `"Paid"`, or `"Deferred"`)
     - `overrideAmount` — optional positive number representing the actual payment amount in USD
   - `incomes` — array of PeriodIncomeEntry subdocuments (default empty), each containing:
     - `income` — required MongoDB ObjectId referencing an Income Document
     - `status` — required PeriodIncomeStatus enum value (`"Pending"` or `"Received"`)
     - `overrideAmount` — optional positive number representing the actual received amount in USD
2. THE Validator SHALL define a Zod schema for Period creation and update requiring `startDate` and `endDate` as valid ISO 8601 date strings; the schema SHALL enforce that `endDate` is strictly after `startDate`; the schema SHALL strip or reject `_id`, `createdAt`, and `updatedAt` if present in the request body.
3. THE Validator SHALL validate `startDate` and `endDate` as non-empty strings that represent valid calendar dates in ISO 8601 format (`YYYY-MM-DD`).
4. THE Validator SHALL use Zod's strict mode to reject Period request bodies containing fields not defined in the Period schema.
5. THE Validator SHALL define the Period Zod schema as a Zod object and export the inferred TypeScript type for use in Controllers and Services.

### Requirement 18: CRUD and Generation Endpoints for Periods

**User Story:** As a Client, I want full CRUD operations and automated batch generation for Periods over HTTP, so that I can manage pay period records and generate them automatically based on active paycheck income schedules.

#### Acceptance Criteria

1. WHEN a `POST /api/periods` request is received with a valid JSON body containing `startDate`, `endDate`, and optional `expenses`/`incomes` arrays, THE Controller SHALL create a new Period Document in the periods Collection and return it with HTTP status `201`.
2. WHEN a `GET /api/periods` request is received, THE Controller SHALL return an array of all Period Documents with HTTP status `200`; each Period SHALL include fully populated `expenses` and `incomes` subdocuments (i.e. the referenced Expense and Income Documents SHALL be embedded in the response, not just their ObjectIds).
3. WHEN a `GET /api/periods/:id` request is received with a valid document ID, THE Controller SHALL return the matching Period Document with HTTP status `200`, with fully populated `expenses` and `incomes` subdocuments.
4. WHEN a `PUT /api/periods/:id` request is received with a valid document ID and a valid JSON body, THE Controller SHALL update the matching Period Document and return the updated Document with HTTP status `200`, with fully populated subdocuments.
5. WHEN a `DELETE /api/periods/:id` request is received with a valid document ID, THE Controller SHALL delete the matching Period Document and return HTTP status `204`.
6. IF a `GET`, `PUT`, or `DELETE` request is received with an ID that does not match any Period Document, THEN THE Controller SHALL return HTTP status `404` with a descriptive error message.
7. IF a route parameter ID is not a valid MongoDB ObjectId format, THEN THE Validator SHALL return HTTP status `400` with a descriptive error message.
8. WHEN a `POST /api/periods/generate/:count` request is received where `count` is an integer between 1 and 12 inclusive, THE Service SHALL generate that many consecutive Period Documents and return them as a collection with HTTP status `200`.
9. WHEN generating periods, THE Service SHALL determine each period's `startDate` using the `dayOfMonth` values of active paycheck Income Documents (where `isPaycheck` is `true` and `inactive` is not `true`), anchoring from the `endDate` of the most recently existing Period (or yesterday if no periods exist).
10. WHEN generating periods, THE Service SHALL set each period's `endDate` to the day before the next period's `startDate`.
11. WHEN generating periods, THE Service SHALL populate the `expenses` array with PeriodExpenseEntry subdocuments for all active Expense Documents (where `inactive` is not `true`) whose `dayOfMonth` falls within the period's date range, each with `status` defaulting to `"Unpaid"`.
12. WHEN generating periods, THE Service SHALL populate the `incomes` array with PeriodIncomeEntry subdocuments for all active Income Documents (where `inactive` is not `true`) whose `dayOfMonth` falls within the period's date range, each with `status` defaulting to `"Pending"`.
13. IF `POST /api/periods/generate/:count` is called with a `count` outside the range [1, 12], THEN THE Controller SHALL return HTTP status `400`.
14. IF `POST /api/periods/generate/:count` is called and no active paycheck Income Documents exist, THEN THE Service SHALL return HTTP status `422` with a descriptive error message.

**User Story:** As a developer, I want the API to apply security headers and request logging globally, so that all responses are hardened against common web vulnerabilities and all requests are logged for observability.

#### Acceptance Criteria
